# Architecture

## Goal

Key Magic exists to solve two related workflow problems on Windows:

1. Prevent destructive or distracting shortcuts from reaching the wrong app.
2. Trigger repeatable text actions without relying on heavyweight desktop automation suites.

The application stays local-first. All keyboard interception, rule storage, and typing automation happen on the user's machine.

## Runtime shape

- KeyMagic.Core contains Windows-specific keyboard models, config persistence, blocking, process discovery, and typing services.
- KeyMagic.Service is the desktop host. It owns the WinForms tray lifetime and starts the local ASP.NET Core API and web dashboard.
- KeyMagic.Web is the React and TypeScript SPA served by the desktop host.
- KeyMagic.Tester is a separate Windows utility for validating keyboard behavior outside the full service.

## Maintainability decisions

- Program.cs now stays thin and delegates composition to KeyMagicRuntime.
- Web dashboard startup is isolated in WebDashboardHost.
- Frontend asset resolution is centralized in FrontendAssetLocator.
- The SPA is organized by feature area instead of a single HTML file.

## Keyboard pipeline

- LowLevelKeyboardHook is the monitoring path. It observes keystrokes for event logging and diagnostics but does not suppress input.
- HotkeyBlocker is the blocking path. It registers foreground-aware `RegisterHotKey` combinations on the WinForms UI thread so matching shortcuts are consumed before they reach the targeted app.
- TypingService is the text-injection path. It registers its own hotkeys on the UI thread and types Unicode text with `SendInput` and `KEYEVENTF_UNICODE`.
- ShortcutBlockingService coordinates the monitoring hook, hotkey blocker, and in-memory event log so the dashboard can show recent blocked and pass-through activity.

## Config and local API

- ConfigStore persists JSON configuration at `%APPDATA%\KeyMagic\config.json`.
- Config reads return cloned snapshots so background threads and UI code do not race over shared mutable state.
- Config writes use a temp-file-plus-replace flow so updates land atomically on disk.
- The local ASP.NET Core API listens only on localhost at the configured dashboard port.
- CORS is limited to `localhost` and `127.0.0.1` for that port.
- Mutating API requests (`POST`, `PUT`, `PATCH`, `DELETE`) require a per-run `X-Admin-Token` generated by WebDashboardHost. The bundled SPA fetches and reuses that token automatically.

## Frontend modules

- status: runtime health and high-level metrics.
- rules: blocking rule creation and management.
- typing: text automation rule creation and triggering.
- events: event stream inspection.
- settings: operational preferences and retention controls.

## Asset resolution and build flow

1. `bun run build` writes the SPA output into `src/KeyMagic.Service/wwwroot`.
2. At runtime, FrontendAssetLocator prefers published `wwwroot`, then local `src/KeyMagic.Service/wwwroot`, then `src/KeyMagic.Web/dist` as a fallback.
3. `dotnet build` and `dotnet publish` include the current `wwwroot` contents without a separate copy step.
4. CI and release validation restore with `--runtime win-x64 /p:SelfContained=true` so self-contained publish steps have the required runtime packs.
5. `dotnet publish` produces self-contained `win-x64` bundles for the service and tester.
